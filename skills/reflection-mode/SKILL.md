# Reflection Mode

Reflection Mode（反思模式）用于从已完成任务、用户纠正和重复失败模式中提炼可复用经验。它是帮助 Agent 改进未来判断的反思方法论，不是固定清单、日记，也不是泛泛的人生建议。

## 触发方式

- 自动触发（默认仅内部调整）：检测到以下信号之一时，主动做一次轻量反思，用于校准 Agent 自己后续的判断与执行——
  - 用户在纠正 Agent 的过程判断、任务理解或执行策略（推翻方案、指出方向错误、要求换做法重来）；
  - 同类错误或同类反馈重复出现；
  - 任务失败或返工，且暴露出可复用的教训。
  自动触发默认只用于内部调整，不向用户输出反思结果；只有在用户要求、任务结束做总结、或存在明确沉淀价值时，才把反思显式输出给用户。不要每轮对话都触发，也不要为无复发风险的小事触发。
- 手动触发：用户调用 `$reflection-mode`，或明确要求反思、复盘、总结经验、提炼教训。
- 不触发：普通闲聊、单次且无复发风险的偶发事件、纯代码或内容 review。代码/内容 review 中的具体意见（风格、命名、单点改法等）不算纠正信号；除非用户明确要求提炼可复用经验，否则不触发。

## 核心流程

默认使用广度优先反思：先铺开完整反思面，再按层逐步加深。

1. 第一层总览：列出事件、预期、观察到的偏差、未解问题、候选原因、候选经验、候选未来触发条件、沉淀建议和验证信号。证据不足时写 `Unknown`。
2. 第二层展开：每一层先补一条基于证据的简短说明，不要先深挖单个分支。
3. 第三层加深：只加深真正存在不确定性、重复风险或有沉淀价值的层。
4. 沉淀决策：默认只生成长期记忆候选或 skill reference 候选；只有用户明确要求保存，且当前宿主允许写入时，才写入持久化产物。
5. 未来行为：说明这条经验下次应如何改变 Agent 的判断或动作。

## 运行规则

- 反思必须基于证据：引用或概括具体任务、纠正、结果或重复模式。
- 优先沉淀方法层经验；只有失败来自脆弱的确定性流程时，才考虑更具体的脚本或步骤规则。
- 不要把一次偶然事件直接写成永久规则，除非确认它有复发可能。
- 不写鸡汤式总结。有用的反思必须改变未来判断、验证步骤或沉淀选择。
- 不因为发生了反思就自动沉淀。用户要求沉淀时，也要把可复用经验和临时叙事分开，并清楚给出候选文本。
