# Reflection Mode

Reflection Mode（反思模式）用于从已完成任务、用户纠正和重复失败模式中提炼可复用经验。它是帮助 Agent 生成可审查经验候选的反思方法论，不会训练模型，也不能保证下次会话自动记住经验；只有被确认并写入宿主 memory、项目文档或 skill reference 等真实加载面后，经验才会影响未来判断。它不是固定清单、日记，也不是泛泛的人生建议。

## 触发方式

- 自动触发（默认仅内部调整）：检测到以下信号之一时，主动做一次轻量反思，用于校准 Agent 自己后续的判断与执行——
  - 用户在纠正 Agent 的过程判断、任务理解或执行策略（推翻方案、指出方向错误、要求换做法重来）；
  - 同类错误或同类反馈重复出现；
  - 任务失败或返工，且暴露出可复用的教训。
  自动触发默认只用于内部调整，不向用户输出反思结果；只有在用户要求、任务结束做总结、或存在明确沉淀价值时，才把反思显式输出给用户。不要每轮对话都触发，也不要为无复发风险的小事触发。
- 手动触发：用户调用 `$reflection-mode`，或明确要求反思、复盘、总结经验、提炼教训。
- 不触发：普通闲聊、单次且无复发风险的偶发事件、纯代码或内容 review。代码/内容 review 中的具体意见（风格、命名、单点改法等）不算纠正信号；除非用户明确要求提炼可复用经验，否则不触发。

## 核心流程

0. 读取候选 inbox：若宿主允许读取文件，优先检查当前项目 `.reflection-mode/pending.jsonl`，只读取与当前项目或明确全局 scope 相关的 `status: pending` 候选；只有处理全局用户偏好或跨项目流程时，才读取 `~/.codex/reflection-mode/pending.jsonl`（Codex）或 `~/.claude/reflection-mode/pending.jsonl`（Claude Code）。读取时按 `last_seen`、`signal_strength` 和 `stale_after_days` 取少量相关记录，不全量塞入上下文。权限未知、读取失败或文件不存在时继续正常反思，不编造历史。
1. 收集证据：概括具体任务、用户纠正、失败/返工结果、重复模式或关键工具选择。
2. 定位偏差：说明预期和实际结果的差异；证据不足时写 `Unknown`，不要强行补结论。
3. 提炼经验：抽象出能改变未来判断、验证步骤或执行策略的可复用经验。
4. 界定适用边界：说明下次什么信号出现时应激活这条经验，什么情况下不适用。
5. 信号分级：按证据次数、用户纠正、失败/返工代价和复发可能标记 `signal_strength: high | medium | low`。
6. 沉淀决策：按“无需保存 / 候选 inbox 待确认 / 宿主授权写入”三档处理；只有达到量化写入标准且当前宿主和用户明确允许写入时，才写入真实加载产物。
7. 元数据、去重与老化：写入或返回候选时，尽量带上 `created_at`、`last_seen`、`evidence_count`、`scope`、`queue_scope`、`project_id`、`canonical_key`、`lesson_hash` 和复查周期。
8. 未来行为：说明这条经验下次应如何改变 Agent 的判断或动作。

## 运行规则

- 反思必须基于证据：引用或概括具体任务、纠正、结果或重复模式。
- 优先沉淀方法层经验；只有失败来自脆弱的确定性流程时，才考虑更具体的脚本或步骤规则。
- 不要把一次偶然事件直接写成永久规则，除非确认它有复发可能。
- 不写鸡汤式总结。有用的反思必须改变未来判断、验证步骤或沉淀选择。
- 候选 inbox 不是长期记忆；pending 记录只表示待确认，不代表未来任务会自动使用。
- 不因为发生了反思就自动沉淀；宿主授权写入至少需要 `signal_strength: high`、`evidence_count >= 2`、结构化证据、明确目标位置、去重元数据和宿主写入权限。写入时只保存可复用经验和必要元数据，不保存临时叙事；不能写入时，清楚给出候选文本和原因。
- 宿主写入能力未知、权限不足，或当前指令要求 memory 只能由用户明确触发时，不得写入，必须退回候选 inbox 待确认。
- 项目相关经验优先使用项目本地 `.reflection-mode/pending.jsonl`；全局用户级队列只用于稳定用户偏好或跨项目流程规则。
- 已保存经验在 `last_seen` 超过复查周期、依赖的仓库/API/流程变化，或出现相反证据时，标记为 `stale_review`，不要继续强化为稳定规则。
