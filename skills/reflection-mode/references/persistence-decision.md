# Persistence Decision

Reflection Mode（反思模式）区分“反思方法”“候选 inbox”和“真正会被加载的持久化产物”。不是每条经验都应该保存，但达到写入标准的经验也不应该长期停留在候选状态。候选队列不是长期记忆；pending 记录只表示待确认，不能当作模型已经学会。

默认行为是先判断写入等级：无需保存、候选 inbox 待确认、宿主授权写入。保存长期记忆、skill reference、项目文档或其他真实加载产物，必须遵守当前宿主的 memory 和文件系统规则；宿主能力未知时，不得写入，只能返回候选和缺失条件。

## 信号强度

每条候选经验先标记 `signal_strength`：

- `high`：当前会话内同类模式至少出现 2 次，且至少有 1 次用户纠正、任务失败、返工或验证暴露出明确代价；或已有持久化候选/日志记录同类模式，本次结构化证据再次确认它仍有效。
- `medium`：只有 1 次清晰证据，但它来自明确用户纠正、共享资产返工、高成本验证遗漏，或未来重新发现成本明显较高。
- `low`：证据单薄、只影响当前对话、缺少行为改变，或主要是 Agent 的猜测。

`signal_strength` 不是情绪评分。它必须能回溯到结构化证据、证据次数、纠正次数、失败/返工代价和复发可能。不要因为反思文字看起来完整就抬高强度。

## 宿主授权写入标准

宿主授权写入必须同时满足以下条件：

1. `signal_strength` 为 `high`。
2. `evidence_count >= 2`，且 `evidence_count` 必须等于结构化 `evidence` 数组长度，并且至少包含一个强信号：用户纠正、同类失败/返工、验证失败、review 发现、或已有候选/日志中的同类记录被本次证据确认。
3. 经验会改变未来判断、验证步骤或执行方式。
4. 经验稳定且有复发可能，未来重新发现成本较高。
5. 目标位置清楚：长期记忆、skill reference、项目文档或其他明确加载产物。候选队列只能作为 inbox，不算已经写入未来行为。
6. 当前宿主允许对应写入，且用户没有表达反对。
7. 具备去重元数据：`queue_scope`、`project_id`、`canonical_key` 和 `lesson_hash`，避免同类经验反复进入队列。

不满足任一条件时，退回候选 inbox 待确认或无需保存。用户明确要求“保存/写入/记住”时可以执行用户授权写入，但仍要检查目标位置、证据和宿主权限；这不属于主动写入。

## 宿主写入权限

“宿主允许写入”必须同时满足：

- 当前系统、开发者或工具说明没有禁止该类写入。
- 目标写入机制真实可用，例如 memory 更新工具、当前 workspace 可写文件、或用户级数据目录写入能力。
- 写入范围和目标符合宿主权限模型；需要审批时已经获得审批。
- 用户没有要求只返回建议、不要保存、不要改文件，或撤回写入授权。

Codex 场景下，如果当前指令规定 memory 只能在用户明确要求时更新，Reflection Mode 不得写入长期记忆；只能返回候选、目标位置和理由。Claude Code 或其他宿主也按同一原则处理：权限未知时退回候选，不能假设允许。

## 元数据

写入或返回候选时，尽量携带这些字段，便于后续老化、审查和效果追踪：

- `created_at`：生成时间，使用 ISO 8601 日期或日期时间。
- `last_seen`：最近一次观察到同类证据的日期或日期时间。
- `evidence_count`：当前会话或可访问日志中的证据次数。
- `correction_count`：用户纠正次数。
- `signal_strength`：`high`、`medium` 或 `low`。
- `experience_type`：`preference`、`repo-contract`、`process`、`review-rule`、`code-pattern` 或 `other`。
- `scope`：适用范围，例如 `global`、repo 路径、skill 名称或具体工具链。
- `queue_scope`：`project` 或 `global`。项目经验默认 `project`；稳定用户偏好或跨项目流程规则才用 `global`。
- `project_id`：项目隔离标识。项目级经验用仓库路径、仓库名或其稳定哈希；全局经验可用 `global`。
- `canonical_key`：同类经验的稳定去重键，例如 `repo:/path:review-rule:diff-first`。
- `lesson_hash`：基于 `scope`、`canonical_key`、`lesson` 和 `next_action` 的短哈希，用于检测重复写入。
- `stale_after_days`：建议复查周期。默认 90 天；依赖 API、团队流程或项目结构的经验用 30 天；稳定风格偏好可用 180 天。
- `applies_to`：后续可用于效果追踪的触发场景列表。

元数据不要求每次都完整，但宿主授权写入至少需要 `created_at`、`last_seen`、`evidence_count`、`signal_strength`、`experience_type`、`scope`、`queue_scope`、`project_id`、`canonical_key` 和 `lesson_hash`。

## 候选 inbox 与跨会话日志

当宿主允许写入文件或用户明确要求维护候选队列时，优先把未确认候选追加到当前项目：

- 项目级：`{project_root}/.reflection-mode/pending.jsonl`

只有稳定用户偏好、跨项目流程规则或用户明确要求全局记录时，才写入用户级目录：

- Codex：`~/.codex/reflection-mode/pending.jsonl`
- Claude Code：`~/.claude/reflection-mode/pending.jsonl`
- 其他宿主：使用宿主公开的 plugin data 目录；未知时不写文件。

每行保存一条 JSON 对象，字段遵循 `output-contract.md` 的 `candidate_record`。候选写入前先用 `canonical_key` 和 `lesson_hash` 检查同项目或同全局 scope 下是否已有同类记录；发现重复时更新候选建议或追加事件，不要无脑新增。使用 `scripts/reflection_queue.py` 列出、确认、跳过、导出、去重和清理候选 inbox。

如果候选最终被确认、跳过、写入或复用，可追加事件到同目录的 `events.jsonl`，不要编造使用历史；较老事件可以滚动归档到 `events.archive.jsonl`。不能写候选队列时，把候选文本和原因返回给用户，避免静默消失。

如果宿主缺少写入能力或路径不可用，必须明确提示：**无法写入候选队列，请手动保存**。

事件保留建议：稳定候选/经验事件默认 `expires_after_days` 为 180；依赖快速变化上下文的短周期事件可用 30。候选 inbox 建议定期清理：默认最多保留 200 条 pending 记录，优先保留当前项目、高 `signal_strength`、最近 `last_seen` 的记录。

## 老化策略

经验不是永久正确。每次准备复用或写入经验时，检查：

- `last_seen` 距今是否超过 `stale_after_days`。
- 经验依赖的仓库、API、团队流程、工具版本或用户偏好是否已经变化。
- 最近任务中是否出现相反证据。

过期或有相反证据时，标记为 `stale_review`，优先提示用户复查、降级为候选或删除；不要在没有确认的情况下继续强化为稳定规则。

## 长期记忆决策

当经验是稳定事实、用户偏好、仓库约定、流程细节或重复出现的项目特定模式，并且应该影响未来对话时，优先考虑长期记忆。若同时达到宿主授权写入标准且宿主允许，就写入长期记忆；否则给出长期记忆候选。

适合作为长期记忆候选的情况：

- 用户反复要求某种回复风格或工作流。
- 仓库有不明显的验证或发布命令。
- 项目存在反复踩坑、重新发现成本高的陷阱。
- 用户纠正暴露出稳定偏好。

如果经验只是猜测、过于宽泛、偏情绪化，或未来任务中很难复用，不要推荐为长期记忆。

## Skill Reference 决策

当经验是可复用方法、决策框架、输出契约或领域流程，并且能指导多类未来任务时，优先考虑 skill reference。若同时达到宿主授权写入标准且宿主允许，就写入 reference；否则给出 skill reference 候选。

适合作为 reference 候选的情况：

- 一套反思流程。
- 一个沉淀决策框架。
- 一个 review 评判标准。
- 一个可重复的项目初始化或发布流程。
- 一种结构化输出格式。
- 一个需要代码片段才能准确表达的 `code-pattern`。

Reference 应该教 Agent 如何思考或行动，不应该保存一次性任务历史。

## 无需保存决策

以下情况推荐不生成持久化产物：

- 普通推理已经显然能得出。
- 只和当前对话有关。
- 基于不完整证据。
- 是一次性偶发事件，复发概率低。
- 只是感受，没有对应行为改变。
- `signal_strength` 为 `low` 且没有用户要求保存。

## 决策问题

推荐或保存前，先回答：

1. 它会改变未来动作或判断吗？
2. 它足够稳定，能经受上下文变化吗？
3. 证据次数、纠正次数和复发成本是否支撑 `signal_strength`？
4. 长期记忆、reference、候选队列、无需保存，哪个是更小、更清晰的处理方式？
5. 宿主授权写入是否比等待确认更能减少重复工作或重复错误？
6. 当前宿主是否明确允许这种写入？

如果答案不清楚，就把它保留为反思记录或候选，不要变成持久规则。

## 写入边界

每次反思都遵守这个边界：

- 如果用户只要求反思，先完成反思和写入等级判断；只有达到宿主授权写入标准且宿主允许时才可写入，否则返回建议或候选。
- 如果用户要求保存，优先执行写入；证据或目标位置不足时，先给出候选文本和目标位置。
- 具体写入时，只保存可复用经验和必要元数据，不保存完整反思叙事，并且只能通过当前宿主支持的 memory 或文件系统机制执行。
- 如果没有可用写入机制或权限，返回候选并说明尚未保存。
- 如果候选仍带有猜测性质，标记为 `provisional`，不要做成稳定规则。示例：`provisional: true`。
