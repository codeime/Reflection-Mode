# Reflective Practice

Reflective Practice（反思实践）把过去经验转化为可审查、可确认的未来行为候选。它适用于任务完成后、用户纠正后、同类错误重复出现时，或 Agent 发现自己被某个先入假设影响结果时。反思本身不会让模型永久学习；只有经验被确认并写入后续会加载的目标位置，才可能影响未来任务。

## 反思方法

反思不是套固定模板，而是把一次任务中的证据转化为未来可复用的判断。先确认事实和偏差，再提炼经验、适用边界和验证信号；证据不足时保留不确定性，不要急着写成永久规则。

1. 事件：发生了什么？
2. 预期：原本应该发生什么？
3. 偏差：现实在哪里偏离了预期？
4. 未解问题：还有什么不知道，不能过早下结论？
5. 候选原因：偏差可能为什么出现？
6. 候选经验：可以抽象出什么可复用原则？
7. 未来触发：下次什么时候应该激活这条经验？
8. 沉淀建议：它应该被推荐为长期记忆候选、skill reference 候选，还是无需保存？
9. 验证信号：以后如何知道这条经验真的产生了帮助？

## 反思深度评分

先估算一个 0-10 的 `reflection_depth_score`，再决定输出细节。评分只用于选择反思深度，不直接决定是否写入。

- `+2`：用户明确纠正了 Agent 的任务理解、策略、优先级或执行方式。
- `+2`：同类问题在当前会话出现至少 2 次。
- `+2`：失败、返工或遗漏影响共享资产、发布流程、仓库规范或用户明确关心的长期结果。
- `+1`：未来重新发现成本高，例如需要特定命令、隐藏上下文、环境约束或项目私有约定。
- `+1`：出现可验证的测试、review、运行时或工具输出证据。
- `+1`：存在可写入的真实加载目标，例如 memory、skill reference 或项目文档。候选队列只算 inbox，不算最终加载目标。
- `+1`：经验可能跨会话复用，而不只是本轮对话的局部修正。

计分边界：

- 多个 `+2` 条件可以同时计分。
- 总分上限封顶为 10。
- 条件只部分满足、证据不清楚或需要推断时，保守取下限；不要为了推动写入而抬高分数。
- 分数只决定反思输出深度，不直接决定 `signal_strength` 或宿主授权写入。

按分数选择深度：

- `0-2`：只做内部轻量校准；通常不输出，不保存。
- `3-5`：输出紧凑反思，给出 `signal_strength` 和沉淀建议。
- `6-8`：输出结构化反思，包含证据、边界、下次动作、元数据和候选文本。
- `9-10`：做完整反思和沉淀决策；若达到宿主授权写入标准且宿主允许，执行写入或候选 inbox 记录。

如果用户明确要求详细复盘，可以提升输出深度；但不能因此跳过证据和写入权限检查。

## 证据来源

- 用户原始请求和后续纠正。
- Agent 的假设、计划、命令、编辑或工具选择。
- 实际输出、测试结果、失败信息或 review 发现。
- 当前会话内重复出现的模式，以及宿主可访问日志或候选 inbox 中与当前项目或明确全局 scope 相关的跨会话重复模式。
- 任何会改变未来行为的明确用户偏好。

## 经验质量

好的经验应该：

- 具体到能指导未来动作。
- 抽象到不只适用于单次事件。
- 基于证据，而不是情绪。
- 足够小，便于记住和复用。
- 明确未来触发条件。

弱经验通常是：

- 没有行动含义的流水账。
- 从一次特殊事件推出的万能规则。
- 对任务结果的重复描述。
- 类似“以后更小心”这种模糊目标。

## 反思深度

低风险任务或经验很明显时，做轻量反思。同类失败重复出现、用户纠正了 Agent、工作影响共享资产，或未来重复犯错成本较高时，增加证据、边界和沉淀判断的细节。

## 效果追踪

当后续任务明确用到了某条已保存经验，并且宿主允许记录事件时，追加一条轻量使用记录，包含经验标识、触发场景、结果、时间和 `outcome_signal`。`outcome_signal` 只能来自真实外部信号，例如 `user_confirmed`、`test_passed`、`review_prevented`，或在没有明确结果时写 `unknown`。不能记录事件时，可以在显式反思输出里写“本次使用了哪条经验”，但不要为了追踪而打断用户当前任务。
