# Output Contract

反思输出应该简洁、基于证据、可复用。除非用户明确要求长篇叙事，否则不要写成流水账。默认使用广度优先方法；只有用户要求结构化反思、任务高风险、问题重复出现，或正在考虑沉淀时，才使用完整分层格式。

## 默认结构

需要结构化反思时，使用这个形状：

```text
第一层总览:
- 事件:
- 预期:
- 观察到的偏差:
- 未解问题:
- 候选原因:
- 候选经验:
- 候选未来触发:
- 沉淀建议:
- 验证信号:

第二层展开:
- 事件: 一条基于证据的简短细节。
- 预期: 原本应该发生什么。
- 观察到的偏差: 现实在哪里偏离预期。
- 未解问题: 还有什么未知，或不应该过度断言。
- 候选原因: 可能根因；必要时标出不确定性。
- 候选经验: 可复用原则，而不是一次性故事。
- 候选未来触发: 下次什么时候应用。
- 沉淀建议: 长期记忆候选、skill reference 候选，或不沉淀。
- 验证信号: 以后如何知道这条经验有效。

第三层加深:
只加深影响未来行为或沉淀选择的层。
```

## 紧凑结构

用户只要快速反思时，使用这个形状：

```text
经验: ...
下次: ...
沉淀建议: 长期记忆候选 | skill reference 候选 | 不沉淀
```

## 沉淀候选

用户要求保存或把反思转成产物时，先输出候选：

```text
候选类型: 长期记忆 | skill reference
候选文本:
...
理由:
...
风险:
...
```

默认不写长期记忆，也不写 skill reference 文件。反思可以推荐沉淀，但保存是单独动作，必须来自用户明确要求，并且当前宿主需要支持对应写入机制。

## 质量门槛

最终输出前检查：

- 是否指出了具体行为改变？
- 是否先广度铺开，再逐层加深？
- 是否避免了模糊建议？
- 是否避免把单次事件过拟合成规则？
- 沉淀建议是否有理由？
- 下次是否能识别并复用这条经验？
